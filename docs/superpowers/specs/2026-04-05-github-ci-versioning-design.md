# GitHub CI/CD, Versioning & Docker Registry

## Context

Teslog Web currently has no CI/CD pipeline, no versioning scheme, and no automated release process. The project deploys via Coolify using the Docker Compose build pack, pulling from the private GitHub repo and building from the Dockerfile. This design introduces GitHub Actions workflows, semantic versioning, Docker image publishing, and branch protection.

## Repos

- **Private** (`steveneppler/teslog-web-private`): All CI/CD workflows run here. Active development repo.
- **Public** (`steveneppler/teslog-web`): Receives mirrored tags and GitHub Releases. Docker images are published under this repo's ghcr.io namespace so they're publicly accessible.

## Workflow 1: CI (`ci.yml`)

**Trigger:** Push to `main`, PRs targeting `main`.

**Steps:**
1. Checkout code
2. Setup PHP 8.3 with extensions: pdo_sqlite, mbstring, zip, bcmath, intl, pcntl
3. Install Composer dependencies
4. Copy `.env.example` to `.env`, generate app key
5. Run `php artisan test`

**Environment:** Ubuntu latest, SQLite (in-memory), no Redis/Docker needed for unit/feature tests.

**Purpose:** Fast feedback on every push and PR. Must pass before PRs can merge.

## Workflow 2: Release (`release.yml`)

**Trigger:** Push of a tag matching `v*` (e.g., `v1.0.0`, `v1.1.0`).

**Steps:**
1. Run the full CI test suite (same as ci.yml)
2. If tests pass:
   - Log in to ghcr.io
   - Build the `app` Docker image from `docker/app/Dockerfile`
   - Tag as `ghcr.io/steveneppler/teslog-web:$VERSION` and `ghcr.io/steveneppler/teslog-web:latest`
   - Push both tags to ghcr.io
3. Create a GitHub Release on the **private** repo with auto-generated release notes (from commits since last tag)

**Docker image scope:** Only the `app` service. The other services (redis, mosquitto, fleet-telemetry, tesla-http-proxy) all use upstream images and don't need to be built/pushed.

**Image visibility:** Published under the public repo namespace (`ghcr.io/steveneppler/teslog-web`) so users can pull without authentication.

## Workflow 3: Mirror (`mirror.yml`)

**Trigger:** Runs after the release workflow completes successfully.

**Steps:**
1. Push the tag to the public repo remote
2. Create a matching GitHub Release on the public repo using the GitHub API, copying the release notes from the private repo release

**Secrets required:** A personal access token (PAT) with `repo` scope for the public repo, stored as a secret in the private repo (e.g., `PUBLIC_REPO_TOKEN`).

## Versioning

- **Scheme:** Semantic Versioning (semver) — `vMAJOR.MINOR.PATCH`
- **Starting version:** `v1.0.0`
- **Release process:** Create and push a tag:
  ```bash
  git tag v1.0.0
  git push origin v1.0.0
  ```
- **Changelog:** Auto-generated by GitHub's release notes feature (groups commits by type based on PR labels)

## Branch Protection (Private Repo)

Applied to `main` branch:

- **Require status checks to pass before merging:** Yes — the CI workflow must pass
- **Require branches to be up to date:** No (avoids rebase churn for a solo developer)
- **Require pull request before merging:** No — direct pushes to main are still allowed
- **Allow force pushes:** No
- **Allow deletions:** No

This is "light" protection — PRs are gated on CI, but the developer can still push directly to main when preferred.

## Secrets to Configure (Private Repo)

| Secret | Purpose |
|--------|---------|
| `GHCR_TOKEN` | GitHub PAT with `write:packages` scope for pushing Docker images to the public repo's ghcr.io |
| `PUBLIC_REPO_TOKEN` | GitHub PAT with `repo` scope for creating releases and pushing tags to the public repo |

Note: `GITHUB_TOKEN` (auto-provided) handles actions on the private repo itself. The additional secrets are only needed for cross-repo operations.

## Coolify Integration

After this is implemented, Coolify should be reconfigured to:
- Pull the pre-built `app` image from `ghcr.io/steveneppler/teslog-web:latest` instead of building from the Dockerfile
- The other services (redis, mosquitto, fleet-telemetry, tesla-http-proxy) continue using their upstream images as defined in docker-compose.yml

This is a Coolify configuration change, not a code change. It can be done after the pipeline is working.

## Files to Create/Modify

| File | Action |
|------|--------|
| `.github/workflows/ci.yml` | Create — CI test workflow |
| `.github/workflows/release.yml` | Create — Build, push image, create release |
| `.github/workflows/mirror.yml` | Create — Mirror tag + release to public repo |

## Verification

1. Push a commit to `main` — CI workflow should run and pass
2. Open a PR — CI should run, branch protection should require it to pass
3. Tag and push `v1.0.0` — release workflow should run tests, build + push Docker image, create GitHub Release
4. Verify Docker image is pullable: `docker pull ghcr.io/steveneppler/teslog-web:v1.0.0`
5. Verify public repo has matching tag and GitHub Release
6. (Later) Reconfigure Coolify to pull from ghcr.io instead of building
